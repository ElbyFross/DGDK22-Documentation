About

Small Tools Hub (STH) its a handful editor tool crated for integration of small tools those need to run in the editor but not deserve its own window to be stated.

The module supplies you with easy to use, still powerful API that allows you to embed a tool into Unity without digging into backstage processes.

How does it work

The tool works as a EditorWindow that envelops Unity services and simplifies them access via own API.

The tool scans entire in-project solutions for STHPlugin derived classes and loads them instances as plug-ins.

Entire messages excluding the OnInspectorGUI calls for each enabled plug-in.
Inspector GUI calls only when the plug-in interface selected at the inspector.

How to..

Start

Small Tools Hub can be started via top editor menu panel by following path:
Doloro GDK \ Small Tools Hub

Warning
On some platforms or package versions the Doloro GDK category can be a sub-category of Tools category.

Enable \ Disable plug-in

Each plugin within Small Tools Hub has a toggle at the left side of its catalog item.

Toggle state defines is the plug-in receives editor messages.

The only callback that works not matter status is, is a OnInspectorGUI over your plug-in, that draws GUI within STH window when the plug-in has been selected in catalog.

That allows you to disable tools you no need at the moment.

Configure a plug-in

To enter the plug-in's GUI you just need to select it in STH's catalog.
After that the right side of the window will draw UI provided with a OnInspectorGUI supplied with the plug-in.

Custom plug-in

Creation

For plug-in create its enough to create a new class derived from the STHPlugin class.
See also: Doloro.EditorExtension.SmallToolsHub.STHPlugin

Warning
The class MUST be located at the editor folder.

The new plug-in class will be automatically found among loaded in-project assemblies and connected to the Small Tools Hub.

using Doloro.EditorExtension.SmallToolsHub;
public class NewPlugin : STHPlugin
{
// Override members using at your plug-in logic here.
}

Cross-session data

For most plug-ins its important to store some data between editor using sessions.
STH tool provides you to fillow a simple pattern to do this.

  1. Create a new class at runtime assembly (not Editor folder).
  2. Derive it from ScriptableObject.
  3. Define members to store. Be sure that private and protected members has SerializeField attribute.
    public ExampleDataContainer : ScriptableObject
    {
    public string prop0
    {
    get => _prop0;
    set => _prop0 = value;
    }
    public int field0;
    [SerializeField] private _prop0;
    }
  4. Declare container class at the plug-in source scope.
  5. Implement DataSave and DataLoad methods as suggested below:
    using Doloro.EditorExtension.SmallToolsHub.Plugins;
    public class NewPlugin : STHPlugin
    {
    // Name of the container asset file to use.
    const string CONTAINER_ASSET_NAME = "ExampleDataContainerFileName";
    public CameraControlDataContainer container
    {
    get
    {
    if (_container == null) DataLoad();
    return _container;
    }
    set => _container = value;
    }
    private CameraControlDataContainer _container;
    public override void DataLoad() =>
    this.DataLoad(ref _container, CONTAINER_ASSET_NAME);
    public override void DataSave() =>
    this.DataSave(ref _container);
    }

Now you data container will be operated automatically by the tool via Doloro GDK cache.
The instance of data container will be always exists by container property call.
